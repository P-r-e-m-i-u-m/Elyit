## Summary

This RFC proposes introducing Redis Streams as an event bus for decoupling service components, enabling better scalability, replay capability and audit logging.

## Motivation

Tight coupling between modules makes independent scaling impossible. No audit trail for state changes. Adding a new consumer requires modifying the producer.

## Detailed Design

Define event schema with eventId, eventType, aggregateId, payload and metadata. Stream naming: user-events, order-events, audit-events. Each service subscribes as a named consumer group.

## Alternatives Considered

RabbitMQ: more mature but additional infrastructure. Kafka: better throughput but significant operational overhead. Direct callbacks: simple but tight coupling remains.
